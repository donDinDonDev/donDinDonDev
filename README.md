## About

I build backend systems for products that need more than code that simply works.

My focus is on systems that remain clear in architecture, reliable in production, and maintainable as complexity grows. I care about business logic, boundaries, integrations, data consistency, testing, and the long-term cost of every technical decision.

A major part of my engineering practice is **Codex-driven development**.

For me, Codex is not a shortcut and not a blind code generator. It is a serious development environment that helps me work with more context, move faster through complex repositories, test ideas carefully, and keep implementation work disciplined.

I have spent hundreds of hours building this workflow in practice: exploring large codebases, decomposing tasks, planning changes, implementing backend features, tightening tests, reviewing architecture, debugging failures, and iterating until the result is clean enough to trust.

I am especially interested in the kind of engineering where human judgment and AI-assisted development reinforce each other: clear thinking, precise context, strong review habits, and fast feedback loops.

## Codex-Driven Development

Codex is now the foundation of my daily development workflow.

I use it to work through real engineering problems: not only writing code, but understanding existing systems, finding the right implementation path, protecting architectural boundaries, improving tests, catching risks, and keeping changes controlled.

My workflow is built around:

- **Deep repository exploration before implementation**
- **Clear task decomposition and execution planning**
- **Test-backed changes instead of blind edits**
- **Careful review of assumptions, risks, and edge cases**
- **Iterative improvement through feedback from tests, CI, and code review**
- **Maintaining context across large and complex codebases**

I believe AI-assisted development is becoming a serious engineering discipline.

The value is not in asking an AI to write code. The value is in knowing how to direct it, constrain it, verify it, and combine it with real engineering judgment.

That is the part I care about most.

---

## Engineering Approach

I prefer practical engineering over decorative complexity.

For me, good backend systems should be easy to reason about, stable in real use, and structured to evolve without turning into chaos.

I care about architecture not as theory, but as a way to make delivery, support, and long-term system evolution more predictable.

My goal is to build software that works, survives change, and can be confidently improved by the next engineer who opens the repository.
